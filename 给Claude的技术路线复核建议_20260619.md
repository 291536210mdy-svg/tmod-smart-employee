## 问题 2：不要在 `run_review_batch()` 中通过 `global ENABLE_LEADERSHIP_PRIORITY` 控制优先级

### 你的技术路线原说法

你建议：

```python
global ENABLE_LEADERSHIP_PRIORITY
ENABLE_LEADERSHIP_PRIORITY = config.enable_leadership_priority
```

并在执行清单中写：

```text

```

### 为什么这里有风险

平台第一版已经设计为多人、多 run、后台线程执行：

```text
RunManager ── ThreadPoolExecutor ── 后台 Worker 线程
```

如果两个 run 同时执行，一个配置 `enable_leadership_priority=True`，另一个配置为 `False` 或未来存在不同业务线配置，那么运行时修改模块级全局变量会产生竞态。

即使第一版前端不暴露关闭优先级的开关，后台测试、dry-run、未来业务线、或者管理员配置都有可能触发这种隐患。  
全局变量在单脚本 CLI 里问题不大，但在服务化、多线程、长任务环境中会成为隐藏状态。

### 我的建议

不要在运行中修改全局变量。更安全的方式是把开关变成函数参数。

建议改成：

```python
def load_leadership_priorities(paths, enabled=True):
    if not enabled:
        return []
    priorities = build_fixed_leadership_priorities()
    ...
```

然后在 `run_review_batch()` 中：

```python
leadership_priorities = load_leadership_priorities(
    config.leadership_priority_paths or DEFAULT_LEADERSHIP_PRIORITIES,
    enabled=config.enable_leadership_priority,
)
```

如果为了兼容旧 CLI 或旧调用，可以保留全局常量作为默认值：

```python
def load_leadership_priorities(paths, enabled=None):
    if enabled is None:
        enabled = ENABLE_LEADERSHIP_PRIORITY
    if not enabled:
        return []
    ...
```

但平台调用必须显式传入 `enabled=config.enable_leadership_priority`。

## 问题 3：内存 EventBus 与 uvicorn 多 worker 部署冲突

### 你的技术路线原说法

你设计了：

```text
EventBus：每 run 一个 dict[run_id] -> list[asyncio.Queue]
```

并在部署章节写：

```text
uvicorn 多 worker 由 Nginx 反代
```

### 为什么这里有风险

内存 EventBus 只在单个 Python 进程内有效。  
如果使用 uvicorn 多 worker，会出现这种情况：

```text
worker A：后台任务正在运行，并往自己的内存 EventBus 推事件
worker B：用户浏览器的 SSE 连接落到这里
```

此时 worker B 的内存里没有 worker A 的实时 queue。  
虽然历史事件可以从数据库重放，但实时事件可能收不到，用户前端会表现为：

- 页面一直停在旧进度；
- 刷新后突然看到新事件；
- SSE 实时性不稳定；
- 多人访问时现象随机。

这是架构层面的确定性问题，不是代码小 bug。

### 可选方案

### 方案 A：第一版明确单进程

如果第一版使用内存 EventBus，那么部署必须明确：

```text
uvicorn workers = 1
RunManager 使用 ThreadPoolExecutor 跑后台任务
Nginx 只反代到这个单进程服务
```

优点：

- 简单；
- 第一版足够；
- 不需要 Redis Pub/Sub；
- 和 ThreadPoolExecutor 方案一致。

缺点：

- 水平扩展能力弱；
- 服务重启会断掉实时队列，但历史事件仍在数据库中。

### 方案 B：SSE 从数据库轮询

EventBus 不依赖内存 queue，而是 SSE 连接每 0.5 秒或 1 秒查一次 `run_events` 表中 `id > last_id` 的新事件。

优点：

- 多 worker 可用；
- 不需要 Redis；
- 实现相对简单。

缺点：

- 实时性稍弱；
- 数据库轮询压力增加，但第一版 run 数量不大，通常可接受。

### 方案 C：Redis Pub/Sub

使用 Redis 作为跨进程事件通道。

优点：

- 更正式；
- 支持多 worker、独立 worker 服务；
- 和后续 Celery/RQ/Arq 路线一致。

缺点：

- 第一版复杂度提高；
- 部署多一个 Redis；
- 代码路径更多。

### 我的建议

如果第一版坚持轻量生产版，我建议：

```text
快速上线版：单 uvicorn worker + 内存 EventBus + DB 历史重放
稍正式版：Redis Pub/Sub + PostgreSQL + 独立 Worker
```

不要在同一个方案里同时写“内存 EventBus”和“uvicorn 多 worker”。

### 请 Claude 判断

请判断第一版到底采用：

1. 单进程内存 EventBus；
2. 数据库轮询式 SSE；
3. Redis Pub/Sub。

并据此修订部署章节、RunManager、EventBus、SSE 端点设计。

## 问题 7：取消任务不能承诺“立即取消”

### 真实代码依据

`call_dify_workflow()` 中使用阻塞的 `requests.post(..., timeout=timeout)`。  
也就是说，如果当前正在调用 Dify Workflow，请求没有返回之前，`should_cancel()` 不会被检查。

你的技术路线建议：

```python
if cancelled(): raise RunCancelled()
```

这是有价值的，但它只能在循环边界生效，例如：

- 开始下一个候选人之前；
- 进入 ranking 之前；
- 进入 reason 之前；
- 进入 export 之前。

它不能中断正在进行中的一个 `requests.post()`。

### 我的建议

取消语义应明确写成：

```text
cancel requested：用户已请求取消
cancelling：系统将在当前候选人处理完成或当前 Dify 请求超时/返回后停止
cancelled：任务已停止，后续候选人不再处理
```

前端提示也应写：

```text
正在取消，当前正在处理的候选人会在本次 Dify 请求结束后停止。
```

不要设计成“点击取消后立即终止所有外部调用”。

如果未来想更强，可以考虑：

- 缩短单次 Dify timeout；
- 使用支持取消的异步 HTTP 客户端；
- 将每个候选人作为独立队列任务；
- Worker 层面进程隔离并允许终止子进程。

但第一版不建议做强杀。

### 请 Claude 判断

请修订 run 状态机、取消接口、前端提示和验收标准，避免承诺“立即取消”。

## 问题 8：QA failed 不应简单等同于 run failed

### 真实代码依据

当前 CLI 行为是 QA 不通过就：

```python
raise SystemExit(...)
```

但这是命令行脚本行为。  
平台化后，run 的状态需要区分：

1. 系统执行失败；
2. Dify 或代码异常；
3. 文件未生成；
4. 流程完整跑完，但 QA 检查不通过。

第四种情况更像是：

```text
run.status = succeeded
run.summary.qa_passed = false
```

而不是：

```text
run.status = failed
```

因为该 run 已经成功生成：

- Excel；
- internal_pack；
- completion；
- qa_report；
- 失败项明细。

QA 不通过是结果质量告警，不一定是系统执行失败。

### 我的建议

状态设计建议：

| 情况 | run.status | qa_passed | 前端展示 |
|---|---|---:|---|
| 所有流程完成，QA 通过 | `succeeded` | `true` | 绿色成功 |
| 所有流程完成，QA 不通过 | `succeeded` | `false` | 黄色或红色 QA 警告 |
| Dify Key 缺失、输入文件无法读取、代码异常、导出失败 | `failed` | `null` 或 `false` | 红色系统失败 |
| 用户取消 | `cancelled` | `null` | 灰色取消 |

可以额外加一个 summary 字段：

```json
{
  "execution_completed": true,
  "qa_passed": false,
  "failed_checks": ["ranking_reason_prefix"]
}
```

### 请 Claude 判断

请明确修订：

- `run_review_batch()` 的返回逻辑；
- `RunManager._execute()` 如何设置状态；
- 前端任务状态展示；
- API 响应中的 `qa` 字段；
