## 2. 逐条判断

### 2.1 “零行为变更” vs 固定优先级修复
**结论：采纳。** “零行为变更”与“启用固定优先级”自相矛盾——`ENABLE_LEADERSHIP_PRIORITY=False` 现状下，启用会改变复宏汉霖/复星健康 4 个奖项组合的排序，这是**明确业务修复**，不是无变更重构。
- 改技术路线章节：§16 阶段标题改“提取 runner + 启用固定优先级业务修复”；删除“dry-run 全量产物与改造前 diff 为空”验收；改为你 §1 列的 7 条验收（非命中奖项排序不变 / 命中奖项槽内重排 / HLX11 别名 / QA 仍生成 / Excel 格式不变）。
- 进一步：把这条**前移为独立“阶段 0”**（见第 4 节），与平台改造解耦，先把业务结果跑对再搭壳。
- Codex 影响：阶段 0 单独提交、单独验证，避免“排序不对”和“平台新代码不稳”混在一起排查。

### 2.2 禁止运行时 `global ENABLE_LEADERSHIP_PRIORITY`
**结论：采纳（关键）。** ThreadPoolExecutor 多 run 并发下，运行时改模块级全局是竞态：两个 run 配置不同 `enabled`、或 dry-run/未来业务线/管理员配置都可能触发。**改为函数参数**，零共享可变全局：
```python
def load_leadership_priorities(paths, enabled=None):
    if enabled is None:
        enabled = ENABLE_LEADERSHIP_PRIORITY      # 旧 CLI/旧调用默认值来源
    if not enabled:
        return []
    priorities = build_fixed_leadership_priorities()
    ...
```
- `run_review_batch()` 显式传：`load_leadership_priorities(paths, enabled=config.enable_leadership_priority)`，**不再 `global`**。
- 阶段 0 同时把模块常量 `ENABLE_LEADERSHIP_PRIORITY = True`（业务修复点），CLI 加 `--disable-leadership-priority` 用于显式关闭。
- 改技术路线章节：§7.2 删 `global` 两行；§17 执行清单删“用 global …”，换“显式传 enabled”。
- Codex 影响：阶段 0 改 `load_leadership_priorities` 签名 + 翻常量；阶段 1 的 `run_review_batch` 按参数传。

### 2.3 内存 EventBus 与多 worker 冲突
**结论：采纳，但选你给的三方案之外的 DB 轮询（你已确认）。** 你 §7 倾向“单进程内存总线”，我建议 **DB 轮询**，理由：
1. worker 本来就把每条事件写入 `run_events` 表（原设计），轮询几乎零额外成本；
2. 多 worker / 服务重启都安全，历史不丢；
3. 省掉“worker 线程→asyncio 事件循环”的 `run_coroutine_threadsafe` 桥接（这是最容易出隐藏 bug 的地方）；
4. 延迟 ~0.75–1s，对“12/38 候选评审中”这种批处理完全够。
- 修订后 SSE 端点：读 `Last-Event-ID`(或 `after_id`)→循环 `SELECT * FROM run_events WHERE run_id=? AND id>? ORDER BY id`，逐条 `yield`，run 到终态且无新行则结束，否则 `await asyncio.sleep(0.75)`。
- **EventBus 简化**：第一版退化为“持久化器”——`publish()` = 写表 + 追加 `events.jsonl`，**不再需要内存订阅队列**。
- **连带修取消**（见 2.7）：取消也改 DB 背书（`runs.cancel_requested` / `status=cancelling`），`should_cancel()` 读库，于是多进程下取消请求落到哪个 worker 都生效。
- 改技术路线章节：§5.3（EventBus 改持久化器 + DB 轮询）、§10（SSE 端点逻辑）、§14（worker 数不再是硬约束，默认 1–2 个均可；Redis Pub/Sub 留给稍正式版）。
- Codex 影响：阶段 3 EventBus 更简单；阶段 5 SSE 端点写成轮询协程；RunManager 的 `should_cancel` 读库。

### 2.4 dataclass 放置位置
**结论：采纳（确实是 bug）。** dataclass 默认值用了 `DEFAULT_TEMPLATE`/`DEFAULT_AWARD_CONFIG`(`review_batch.py:23-25`)，若放在常量定义前会 `NameError`。双保险：
1. dataclass 放在**全部模块常量之后**（`DEFAULT_AWARD_CONFIG_DATA` 之后）；
2. `template_path`/`award_config_path` 默认 `None` + `__post_init__` 兜底：
```python
@dataclass
class ReviewBatchConfig:
    input_path: Path
    output_dir: Path
    template_path: Path | None = None
    award_config_path: Path | None = None
    ...
    def __post_init__(self):
        if self.template_path is None: self.template_path = DEFAULT_TEMPLATE
        if self.award_config_path is None: self.award_config_path = DEFAULT_AWARD_CONFIG
```
- 改技术路线章节：§7.1 把“放 imports 之后”改“放全部常量定义之后”，并补 `None`+`__post_init__`。
- Codex 影响：阶段 1 第 1 步位置说明更正。

### 2.5 viewer 权限过松
**结论：采纳（你已确认保守只读）。** 评优数据含姓名/推荐人/申报理由/评审分/风险标记，最终 Excel 仍属敏感结果。修订权限表：

| 能力 | viewer | reviewer | admin |
|---|---:|---:|---:|
| 登录 | ✓ | ✓ | ✓ |
| 看被授权 run 的状态/进度/QA结论 | ✓ | ✓ | ✓ |
| 看所有 run | ✗ | ✓ | ✓ |
| 创建 run | ✗ | ✓ | ✓ |
| 取消 run | ✗ | 本人/全部 | ✓ |
| 下载最终 Excel / 待补充清单 | ✗ | ✓ | ✓ |
| 看候选人详情 | ✗ | ✓ | ✓ |
| 下载 internal_pack / qa_report | ✗ | ✓ | ✓ |
| 建账号/改角色/清理产物 | ✗ | ✗ | ✓ |

viewer 第一版=只读旁观（仅状态，无任何下载）。“普通用户也能下载最终结果”留作后续可配置项。
- 改技术路线章节：§13 权限表替换；§11 前端按角色隐藏“新建任务”入口与下载按钮（viewer 不可见）。
- Codex 影响：阶段 7 `require_role`：创建/下载/详情 ≥reviewer；阶段 6 前端按 `role` 渲染。

### 2.6 SQLite / PostgreSQL 口径统一
**结论：采纳（你已确认 PG 为正式路径）。** 统一口径：
```
本地开发/测试：SQLite（零依赖）
服务器/团队第一版：PostgreSQL（正式路径）
SQLAlchemy 模型必须同时兼容二者
docker-compose 默认带 PostgreSQL
```
- SQLite 专属 PRAGMA（WAL）只在 `database_url` 为 sqlite 时设置（`event.listens_for(engine,"connect")` 内判断 dialect）；PG 不执行。
- 提供 `.env.example.local`(SQLite) 与 `.env.example.server`(PostgreSQL) 两份。
- 改技术路线章节：§8（模型可移植 + PRAGMA 条件化）、§14（compose 默认 PG，两份 .env）。
- Codex 影响：阶段 2 `db/base.py` 按 dialect 分支；阶段 7 compose 加 `db: postgres:16`。

### 2.7 取消任务不能承诺“立即取消”
**结论：采纳。** `call_dify_workflow` 用阻塞 `requests.post(..., timeout)`，`should_cancel()` 只能在循环边界生效，**无法打断进行中的 Dify 请求**（最多等其返回或 timeout）。语义明确为：
```
cancel_requested：用户已请求取消
cancelling：将在当前候选处理完成 / 当前 Dify 请求返回或超时后停止
cancelled：已停止，后续候选不再处理
```
前端提示：“正在取消，当前候选会在本次 Dify 请求结束后停止。”
- 状态机加 `cancelling`：`running → cancelling → cancelled`。取消接口返回 `status:"cancelling"`。
- 第一版不做强杀；未来可缩短 timeout / 用可取消异步客户端 / 候选级队列任务。
- 改技术路线章节：§8 状态机加 `cancelling`；§9 cancel 响应；§11 前端文案；§15 验收（不承诺“立即取消”，验“请求取消后在边界停止”）。
- Codex 影响：阶段 3 `should_cancel` 读 `runs.cancel_requested`；阶段 5 cancel 接口设标志；阶段 6 文案。

### 2.8 QA failed ≠ run failed
**结论：采纳。** 流程跑完但 QA 不过，产物(Excel/internal_pack/completion/qa_report)都已生成，是**质量告警**不是系统失败。状态映射：

| 情况 | run.status | qa_passed | 前端 |
|---|---|---:|---|
| 跑完 + QA 通过 | `succeeded` | `true` | 绿 |
| 跑完 + QA 不过 | `succeeded` | `false` | 黄/红 QA 警告 |
| Dify Key 缺失/读文件失败/代码异常/导出失败 | `failed` | `null` | 红 系统失败 |
| 用户取消 | `cancelled` | `null` | 灰 |

`run_summary.json` / `Run.summary_json` 加：
```json
{"execution_completed": true, "qa_passed": false, "failed_checks": ["ranking_reason_prefix"]}
```
- `run_review_batch()` 已按 §0 改为**不 sys.exit**、返回 `qa_passed`——与本条一致，无需再改返回逻辑。
- `RunManager._execute()`：正常返回→`succeeded`(无论 qa_passed)；抛异常→`failed`(记 error_message)；`RunCancelled`→`cancelled`。
- 改技术路线章节：§7（确认不 sys.exit）、§8（summary 字段）、§9（`qa` 字段语义）、§11（三色展示）、§15（验收区分 succeeded+qa_false 与 failed）。
- Codex 影响：阶段 4 `_execute` 状态映射；阶段 5 API `qa`；阶段 6 状态徽标。

## 3. 修订后的第一版架构约束（硬约束，Codex 必须遵守）

1. **优先级开关只走参数**，禁止运行时改 `ENABLE_LEADERSHIP_PRIORITY`；模块常量翻 `True` 仅作 CLI/旧调用默认。
2. **事件与取消都以数据库为单一事实源**：事件写 `run_events`，SSE 轮询读；取消写 `runs.cancel_requested`，worker 边界读。无内存订阅队列、无跨线程事件循环桥接。
3. **后台执行 = ThreadPoolExecutor**（每进程一个）；worker 数非硬约束（DB 轮询/DB 取消使多 worker 也安全），第一版默认 1–2 个。
4. **数据库**：本地 SQLite、服务器 PostgreSQL，模型可移植，PRAGMA 条件化。
5. **权限最小化**：创建/下载/详情 ≥reviewer；viewer 只读无下载；账号/清理 admin。
6. **状态机**：`created → queued → running → succeeded | failed | cancelled`，外加过渡态 `cancelling`；QA 结果走 `summary.qa_passed`，不进主状态。
7. **取消是协作式**，不打断在途 Dify 请求；对外措辞与验收都不承诺“立即取消”。
8. **review_batch.py 业务逻辑零重写**：阶段 0 只翻开关+加参数，阶段 1 只提取函数+插事件回调。
