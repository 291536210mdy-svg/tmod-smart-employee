# 第一版技术路线：评优拟推荐名单生成与解释智能体

## 概要

第一版做成“可复用产品化后端骨架”，不是一次性脚本。核心目标是：读取现有申报 Excel 和评优规则 PDF，调用 Dify Workflow 做证据理解与结构化输出，本地 Python 负责打分、排序、生成内部评审包，最终导出你给的 `评选结果输出格式.xlsx` 这种名单表。

第一版先默认每个奖项导出 Top 2，等整体跑通后改成人工配置名额。最终 Excel 不展示分数、置信度、权重，只展示评委/业务方需要看的拟推荐名单。

## 技术路线

- 本地 Python 做产品化工程骨架：`parser / dify_client / scoring / ranking / export / audit` 分层，后续可直接套 FastAPI。
- Dify 做企业内模型能力：规则检索、申报理由理解、证据强弱判断、团队负责人/成员抽取、事迹改写、解释生成。
- 本地 Python 不让 LLM 直接决定获奖人：LLM 只输出结构化证据等级，本地规则引擎根据 `award_config` 计算内部推荐分。
- 输出分两层：
  - 内部评审包：JSON/JSONL，含证据等级、规则命中、缺失证据、风险提示、排序、解释、来源追踪。
  - 最终名单 Excel：严格按现有模板字段输出：`序号 / 奖项名称 / 主体 / 所属BU / 提报人 / 团队负责人 / 团队成员 / 事迹`。
- 第一版入口建议保留 CLI：`run_review --input sample_input.xlsx --rules 2025年度评优说明.pdf --output 评选结果输出.xlsx`。
- 同时按 FastAPI 思路设计接口，后续直接补 `/upload_excel`、`/run_review`、`/candidates`、`/export_report`，不重构核心逻辑。

## 关键模块

### Excel 解析模块

- 读取现有固定字段，不新增字段。
- 每行生成一个 `candidate_id`。
- 保留原始行 JSON，作为后续审计、解释和问答的基础数据。

### PDF 规则结构化模块

- 把 25 个奖项、通用原则、一票否决、评优口径整理成 `rules.json`。
- 将结构化规则导入或同步到 Dify 知识库。

### Dify Workflow

输入：

- 候选人信息
- 奖项名称
- 脱敏申报理由
- 完整申报理由
- 规则上下文

输出：

- 证据等级
- 规则命中
- 缺失证据
- 风险点
- 负责人/成员抽取
- 事迹摘要
- 解释文本

### 打分排序模块

- 规则命中、量化程度、价值影响、创新性、战略契合度由 Dify 给等级。
- Python 根据配置换算内部得分。
- 一线/海外倾向只做同分排序依据，不直接加分。
- 风控或重大缺证不自动入选，只进入人工复核。

### 导出模块

- 最终 Excel 只输出 Top 2 拟推荐名单。
- 团队负责人/成员无法可靠抽取时填 `待补充`。
- 另生成 `待补充清单.xlsx` 和 `internal_review_pack.jsonl`，方便人工复核。

## 8 个高优先点落地

- 内部分数只存在评审包，不进入最终 Excel。
- LLM 不直接打总分，只判断证据等级。
- 权重放在 `award_config`，不硬编码。
- 一线/海外只作为同等条件下的排序倾向。
- 最终 Excel 每个字段在内部包里记录来源、抽取方式、置信度。
- 缺负责人/成员时填 `待补充`，并输出补充清单。
- 分两条 LLM 路径：评分路径脱敏，字段抽取路径使用完整文本。
- 系统输出叫“拟推荐名单”，最终结果仍由评审会确认。

## Dify 配置情况

我看了 `E:\Agent学习\my_project\.env`，已有这些基础项：

- `DIFY_BASE_URL`
- `DIFY_API_KEY`
- `DIFY_USER`
- LLM 模型配置

还需要后续从 Dify App 里确认或补充：

- `DIFY_API_KEY` 是否就是“评优评审 Workflow App”的 API Key。
- 规则知识库 Dataset ID，或确认规则 PDF 由人工先上传到 Dify 知识库。
- 如果后面做对话框，还需要单独的“评优结果问答 Chatflow”API Key。
- Dify Workflow 的输入变量名和输出 JSON 字段名，第一版要固定下来，避免 Python 调用时不稳定。

## 第一版验收标准

- 能读取当前申报 Excel，不要求新增字段。
- 能基于 2025 评优规则生成每个候选人的内部评审包。
- 能按奖项分组排序，默认每个奖项输出 Top 2。
- 能生成与你给的 `评选结果输出格式.xlsx` 字段一致的最终名单表。
- 能标记缺证、风险、负责人/成员待补充。
- 任意一个候选人都能追溯：为什么推荐、命中了哪些规则、哪些证据不足、为什么排在另一个人前面。

## 默认假设

- 第一版先不做前端 UI。
- 第一版先不做自然语言问答框，但内部评审包要为问答框预留证据和解释字段。
- 第一版先按 Top 2 出名单，后续改成每个奖项人工配置名额。
- 最终名单是“拟推荐名单”，不是自动宣判获奖结果。
